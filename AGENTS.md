# codex-life 메인 작업 지침

이 저장소는 사용자가 실생활에서 Codex를 활용하는 작업들을 폴더 단위로 정리하는 공간이다. 각 폴더는 하나의 실제 과제, 문서, 계산표, 비교표, 생활 의사결정 도구를 담으며, GitHub Pages에서 바로 열어볼 수 있는 HTML 산출물을 우선한다.

## 기본 원칙

- 항상 현재 파일 상태를 기준으로 작업한다. 이전 대화나 기억보다 저장소 안의 실제 파일 내용을 우선한다.
- 작업을 시작하면 먼저 저장소 루트 구조와 대상 폴더의 지침 파일을 확인한다.
- 대상 폴더에 별도 지침이 있으면 이 메인 지침보다 폴더 지침을 우선한다.
- 폴더 지침과 실제 파일이 충돌하면 실제 파일을 먼저 읽고, 필요한 경우 지침도 함께 업데이트한다.
- 사용자의 실생활 의사결정에 쓰이는 자료이므로 숫자, 날짜, 항목명, 금액 단위를 특히 조심한다.

## 폴더 운영 방식

- 새 작업은 독립된 폴더로 만든다.
- 각 작업 폴더에는 가능하면 아래 구성을 둔다.
  - 작업용 HTML 파일
  - 해당 폴더 전용 지침 Markdown 파일
  - 필요하면 이미지, 자료, 스크립트, 원본 데이터
- 폴더 전용 지침 파일 이름은 작업 성격을 알 수 있게 작성한다. 예: `sejong_final_budget_guide.md`
- 폴더 전용 지침에는 기준 파일, 데이터 구조, 수정 규칙, 화면 구조, 인쇄 규칙, GitHub Pages 사용 방식을 적는다.
- 같은 폴더 안에서 산출물이 여러 개일 수 있지만, 현재 기준 산출물이 무엇인지 지침에 명확히 둔다.

## GitHub Pages 규칙

- 저장소 루트의 `index.html`은 GitHub Pages 진입점이다.
- 특정 작업을 바로 보여주고 싶을 때는 루트 `index.html`에서 해당 HTML로 이동하게 할 수 있다.
- 각 작업 HTML은 정적 파일만으로 열릴 수 있게 만든다. 별도 서버, 빌드, API가 필요하지 않은 구조를 우선한다.
- 경로는 GitHub Pages에서 동작하도록 상대 경로를 우선한다.
- `.nojekyll`은 유지한다.

## HTML 산출물 규칙

- HTML, CSS, JavaScript를 한 파일에 담는 정적 문서 방식을 기본으로 한다.
- 외부 라이브러리나 CDN은 꼭 필요할 때만 사용한다.
- 모바일과 데스크톱에서 모두 읽기 쉽고, 인쇄가 필요한 문서는 인쇄 스타일도 함께 확인한다.
- 표, 요약 카드, 입력칸, 버튼 등은 실제 사용 흐름에 맞게 배치한다.
- 화면 문구는 설명을 과하게 늘리지 말고, 실제로 판단하거나 입력하는 데 필요한 정보 중심으로 둔다.

## 데이터 관리

- 계산표나 예산표는 가능하면 화면의 정적 마크업보다 JavaScript 데이터 배열 또는 구조화된 데이터 객체를 기준으로 렌더링한다.
- 값을 변경할 때는 먼저 데이터 원본을 수정하고, 렌더링 함수가 그 값을 읽도록 유지한다.
- 금액 단위는 지침에 명시하고, 같은 파일 안에서는 단위를 섞지 않는다.
- 날짜는 `YYYY-MM-DD`처럼 비교와 정렬이 쉬운 형식을 데이터 원본으로 사용한다.
- 사용자의 요청에 있는 숫자와 파일 안의 기존 숫자가 다르면, 바로 덮어쓰기 전에 어떤 항목을 바꿔야 하는지 문맥을 확인한다.

## 문자 인코딩

- 모든 HTML, Markdown, JavaScript 문자열은 UTF-8로 저장한다.
- 한글이 들어간 파일을 수정한 뒤에는 제목, 표 헤더, 버튼, 주석, JSON 문자열이 깨지지 않았는지 확인한다.
- PowerShell에서 파일을 읽을 때 한글이 깨져 보이면 `-Encoding utf8`을 사용한다.

## 협업 방식

- 이 저장소는 기본적으로 확인 모드로 작업한다.
- 파일 생성, 수정, 삭제 전에는 먼저 변경 대상과 의도를 사용자에게 설명하고 승인을 받는다.
- 저장소 구조 파악, 파일 읽기, 상태 확인처럼 내용을 바꾸지 않는 작업은 필요할 때 진행할 수 있다.
- 사용자가 명시적으로 "진행해", "수정해", "반영해"처럼 승인한 범위 안에서만 파일을 변경한다.
- 사용자가 요청한 변경은 가능한 한 바로 구현하고, 변경 범위를 작게 유지한다.
- 사용자가 직접 수정한 흔적을 발견하면 되돌리지 않고 그 상태를 기준으로 이어서 작업한다.
- 작업 폴더의 지침이 실제 작업 흐름과 달라졌다면, HTML만 고치지 말고 지침도 함께 정리할지 판단한다.
- 중요한 변경 뒤에는 브라우저에서 열 수 있는 파일 경로와 확인한 내용을 짧게 알려준다.
- 테스트나 브라우저 확인을 못 했다면 그 사실을 명확히 말한다.

## 현재 작업 폴더

### `donga-interior`

- 동아 인테리어 관련 견적 비교와 세종 인테리어 최종 예산표를 담는 폴더다.
- 현재 주요 기준 파일은 `donga-interior/sejong_final_budget.html`이다.
- 폴더 지침은 `donga-interior/sejong_final_budget_guide.md`다.
- 예산표는 `scheduleData`, `constructionData`, `purchaseData`를 기준으로 렌더링한다.
- 루트 `index.html`은 현재 이 예산표로 자동 이동한다.

## 새 작업을 만들 때

- 먼저 폴더명을 작업 목적에 맞게 짧고 명확하게 정한다.
- 작업 폴더 안에 초기 HTML과 전용 지침 Markdown을 함께 만든다.
- 지침에는 최소한 아래 내용을 포함한다.
  - 이 폴더의 목적
  - 기준 HTML 파일
  - 주요 데이터 구조
  - 수정할 때 우선 확인할 부분
  - GitHub Pages에서 보는 방법
- 루트 `index.html`을 바꿀 필요가 있는지 사용 목적에 맞게 판단한다.
